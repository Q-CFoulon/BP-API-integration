Executive Summary

The enterprise model integrates Spyglass MDR and SecOps services with Blackpoint API and customer Defender XDR tenants, replacing manual portal-hopping with a unified case, response, and AI-assisted triage surface. Design Overview

  • Pre-authentication and Conditional Access are enforced at the Entra ID edge.
  • App and data services run behind private networking and private DNS resolution.
  • Dev and Prod are segregated across subscriptions, VNets, connector groups, and enterprise apps.
  • PaaS-first architecture is instrumented for policy, monitoring, and compliance evidence from day one.

High-Level Architecture

Logical Architecture Summary

  • Identity Edge: Entra ID with App Proxy cloud service.
  • Hub Layer: NAT Gateway and private DNS.
  • Spoke Layers: separate Dev and Prod connectors and private workloads.
  • Workloads: App Service, Function Apps, SQL or PostgreSQL, Blob Storage, Key Vault.
  • Security and Monitoring: Defender for Cloud, App Insights, Log Analytics, Event Hub forwarding.

Traffic Flow

  1. User request lands on App Proxy published endpoint.
  2. Entra ID evaluates authentication and Conditional Access policy gates.
  3. App Proxy relays authenticated requests to available connector over outbound session.
  4. Connector forwards to App Service through Private Endpoint.
  5. Application calls data and secret services with managed identities and private connectivity.
  6. Outbound traffic exits through hub NAT for controlled egress.

User and Load Assumptions

  • Platform users (application consumers): 25-75 concurrent analysts within the Quisitive SecOps team, split between onshore (US) and offshore (India) staff. These are the only users placing direct load on application services (App Service, Function Apps, dashboards).
  • Monitored tenant end-users: Each managed tenant contains 1K-10K end-users generating logs, alerts, and security activity. These users do not interact with the platform directly; their activity is ingested via API polling and event-driven pipelines without creating application-tier session pressure.
  • Scaling model: As the number of managed tenants grows, the Quisitive SecOps team scales onshore and offshore headcount proportionally to match operational demand. Platform user count remains bounded to the analyst team size, not the monitored user population.
  • Load implication: Application compute, session, and concurrency sizing targets 25-75 simultaneous users. Ingestion and data-plane sizing targets the aggregate log and alert volume from monitored tenants (driven by tenant count and per-tenant user activity, not by platform user sessions).
